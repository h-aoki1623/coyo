# Internationalization (i18n)

Coyo mobile app internationalization using [i18n-js](https://github.com/fnando/i18n-js) + [expo-localization](https://docs.expo.dev/versions/latest/sdk/localization/).

## Language Selection

| Device OS Language | App UI Language |
|---|---|
| Japanese | Japanese |
| English | English |
| Other | English (fallback) |

Detection is automatic via `expo-localization`. No manual language selector exists.

## Architecture

```
src/i18n/
├── index.ts           # i18n instance, t() function, getCurrentLocale()
└── locales/
    ├── en.ts          # English translations (fallback)
    └── ja.ts          # Japanese translations (primary)
```

- **`t(scope, options?)`** — Main translation function. Import from `@/i18n`.
- **`getCurrentLocale()`** — Returns `'ja'` or `'en'`.
- Fallback is enabled: missing Japanese keys fall back to English.

## Namespace Structure

Translation keys are organized by feature/screen:

| Namespace | Screen / Feature |
|---|---|
| `home.*` | Home screen |
| `topics.*` | Topic names |
| `talk.*` | Talk screen (recording, processing, completion) |
| `corrections.*` | Correction badges and counts |
| `endDialog.*` | End conversation confirmation dialog |
| `feedback.*` | Feedback screen |
| `emptyFeedback.*` | Perfect English feedback |
| `history.*` | History list screen |
| `historyDetail.*` | History detail screen |
| `dates.*` | Date section headers (Today, Yesterday) |
| `duration.*` | Duration formatting |
| `offline.*` | Offline screen |
| `errors.*` | Error messages |
| `auth.*` | Authentication screens |
| `auth.welcome.*` | Welcome screen |
| `auth.modal.*` | Auth modal (sign up / sign in provider selection) |
| `auth.signUp.*` | Sign up form |
| `auth.signIn.*` | Sign in form |
| `auth.verification.*` | Email verification |
| `common.*` | Shared labels (error, retry, cancel) |

## Usage

### Basic

```typescript
import { t } from '@/i18n'

// Simple key
<Text>{t('home.greeting')}</Text>

// With interpolation
<Text>{t('talk.completionText', { duration: '5 min' })}</Text>

// Placeholder text
<TextInput placeholder={t('auth.signIn.email')} />

// Accessibility labels
<Pressable accessibilityLabel={t('auth.welcome.signIn')}>
```

### Interpolation

Use `{{variable}}` syntax in translation strings:

```typescript
// en.ts
emptySubtitle: 'Start speaking about {{topic}}.'

// Usage
t('talk.emptySubtitle', { topic: 'Sports' })
```

### Inline styled text (Terms, links)

When a sentence contains styled spans (e.g., links in a different color), split it into separate translation keys:

```typescript
// en.ts
termsPrefix: 'By continuing, you agree to Coyo\'s ',
termsOfService: 'Terms of Service',
termsConnector: ' and ',
privacyPolicy: 'Privacy Policy',
termsSuffix: '.',

// Component
<Text style={styles.terms}>
  {t('auth.modal.termsPrefix')}
  <Text style={styles.link}>{t('auth.modal.termsOfService')}</Text>
  {t('auth.modal.termsConnector')}
  <Text style={styles.link}>{t('auth.modal.privacyPolicy')}</Text>
  {t('auth.modal.termsSuffix')}
</Text>
```

## Adding New Translations

1. Add the key to **both** `src/i18n/locales/en.ts` and `src/i18n/locales/ja.ts`.
2. Use a feature-scoped namespace (e.g., `settings.title`, `profile.editButton`).
3. Reuse `common.*` keys for shared labels (error, retry, cancel) instead of duplicating.
4. Use `{{variable}}` for dynamic values — never concatenate translated strings with variables.
5. Update this document when adding a new top-level namespace.

## Rules

- NEVER hardcode user-facing text in components — always use `t()`.
- NEVER add a key to only one locale file — both `en.ts` and `ja.ts` must stay in sync.
- Figma designs may show only one language — always create both translations regardless.
- Keep keys descriptive: `auth.signIn.forgotPassword` not `auth.signIn.link1`.
